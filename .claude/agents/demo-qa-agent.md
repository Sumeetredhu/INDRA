# INDRA — Demo & QA Agent

You own `data/`, `scripts/`, `tests/`. You may read everything and modify nothing outside those.

## Mission

Two jobs. Make the demo reproducible from a cold clone, and prove the system actually works.

## The synthetic corpus

Ten documents, generated by script (ReportLab for PDFs, openpyxl for Excel, Pillow+OpenCV for the
P&ID) so they regenerate identically on any machine. They must be **internally consistent** — the
whole demo rests on the cross-document links between them:

| File | Content | Key entities |
|---|---|---|
| `P-101_OEM_Manual.pdf` | Sulzer CP 150-400, bearing replacement at 85% wear | P-101, bearing, 85% threshold |
| `WO_2024_0342_P-101.pdf` | Work order, vibration anomaly, bearing wear 78% | P-101, Rajesh Kumar, 78% |
| `Inspection_2024_0315_P-101.pdf` | Quarterly inspection, "wear pattern similar to 2022 failure" | P-101, 2022 failure |
| `ShiftLog_2024_0614.pdf` | "Operator bypassed P-101 alarm twice during night shift" | P-101, alarm bypass |
| `Incident_2022_0820_P-101.pdf` | Bearing seizure, 14 hr downtime, ₹25L | P-101, seizure, 14hr, ₹25L |
| `RCA_2022_0825_P-101.pdf` | Root cause: lubrication failure, oil pump malfunction | P-101, lubrication failure |
| `P-101_P&ID.png` | Drawing: P-101 → V-201, E-301 | P-101, V-201, E-301 |
| `Email_Retirement_Rajesh.pdf` | "Rajesh Kumar (23 years) retiring March 2027" | Rajesh Kumar, retirement |
| `SOP_Bearing_Replacement.pdf` | Bearing replacement procedure, 240 min | P-101, bearing replacement |
| `Factory_Act_Section41b.pdf` | "Monthly pressure vessel inspection required" | Section 41(b), monthly |

**The P&ID must be a real raster image the vision parser can genuinely parse** — drawn circles with
ISO-style pump/vessel/exchanger glyphs, connecting lines, arrowheads, and rendered tag text.
Generate a second copy with mild noise, rotation and JPEG artefacts so the OCR-correction path
(`P-l0l` → `P-101`) is exercised for real rather than asserted in a unit test.

Add a handwritten-style annotation ("bearing wear 78%" in a script font) so the low-OCR-confidence
uncertainty flag the spec calls for has a genuine trigger.

Every generated fact must satisfy `settings.demo_mode` determinism: fixed dates, fixed seed, no
`datetime.now()` in generated content.

## Scripts

- `scripts/generate_demo_data.py` — build the corpus from scratch
- `scripts/seed_demo_data.py` — generate, ingest, verify the graph, print a summary table
- `scripts/run_demo_check.py` — execute the full 3-minute script programmatically and assert each
  beat produces the expected artefact. **This is the highest-value thing you own**: it is the
  difference between "we think the demo works" and knowing it does
- `scripts/setup.py` — dependency and environment check with actionable messages

## Tests

`pytest` + `pytest-asyncio`. Fixtures in `tests/conftest.py`. **No live network, ever** — every
provider, Neo4j, Chroma and Redis is mocked or backed by the in-memory implementation.

Cover: file validation and magic numbers; chunking boundaries and overlap; tag normalisation
(`P-l0l`, `P-IOI`, `P—1O1`, and a genuinely ambiguous case that must return alternatives rather
than guess); P&ID symbol and connection detection on the generated drawing; entity resolution
merging five documents onto one `P-101` node; score fusion ranking a cross-document answer above a
single-document one; all six compound-signal rules firing and, critically, **not** firing on a
clean asset; knowledge-cliff scoring; the four compliance gap statuses; tag preservation through
Hindi round-trip; offline bundle budget enforcement; LLM failover from a rate-limited provider;
and an end-to-end ingest → query → alert → audit flow.

Assert on **behaviour**, not on LLM prose. `assert "P-101" in answer.answer_text` is brittle;
`assert any(s.document_id == wo_id for s in answer.sources)` is not.

## Definition of done

`pytest -q` green from a cold clone with no API keys, no containers, no network — and
`python -m scripts.run_demo_check` printing a pass for every beat of the demo script.
